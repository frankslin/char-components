# 重構計畫

本文是給人看的專案重構規劃，說明「為什麼要做」「打算怎麼做」「哪些地方不能動」。給 AI agent 執行時遵守的具體規則與任務拆解，見專案根目錄的 `AGENTS.md`；本文著重在決策脈絡與取捨，方便日後回顧「當初為什麼這樣決定」。

背景與現有原始碼的技術分析見 [04-開發理解與重構指引.md](./04-開發理解與重構指引.md)，建議先讀那篇再讀本文。

## 為什麼要重構

`部件檢索.htm` 目前是一個 4.3MB 的單一檔案，其中約 2.4MB 是壓成一行的 JavaScript 陣列字面量（漢字拆分資料）。這個形式完全符合原作者在 [03-部件檢索公開授權.md](./03-部件檢索公開授權.md) 裡的設計目標——「核心程式獨立壓縮成一行」「一個 html 檔放上網站就完成部署」——對**終端使用者**與**想嵌入自己網站的站長**來說，這是對的形式。

但對**開發者**來說，這個形式有明顯代價：

- 資料與邏輯混在一起，`git diff` 完全無法閱讀，每次拆分表更新（作者提到會持續花數月核對）都是一次不可審閱的黑箱變更。
- Unicode 版本更新時（15.1 → 17.0，作者已經歷過一次），`GetBlock`/`GetIndex` 裡手刻的十六進位偏移量要手動重新推導，容易出錯又難以驗證。
- 資料本身（漢字拆分樹、異體映射）其實是獨立於這個 HTML UI 的知識資產，鎖死在這個檔案格式裡，其他工具想重用都得自己重新解析。

重構的目標，是把「開發時的原始碼結構」和「使用者拿到手的產出物」分開：**開發者看到的是乾淨、可 diff、可測試的多檔案結構；使用者拿到的仍然是同一個零依賴、可離線使用的單一 HTML 檔。** 這不是取捨，而是兩者可以並存——用一個建置腳本把前者組裝成後者即可。

## 三個階段，以及為什麼是這個順序

### 階段一：資料抽離

**做什麼**：把 `dt`（拆分表）、`rt`（展示異體表）、`vt`（比對用單向映射表）從內嵌的一行 JS 陣列，改成獨立的資料檔（例如一字一行的 JSON Lines），並把 `GetBlock`/`GetIndex` 裡手刻的 Unicode 區塊偏移量改成從區塊列表生成，而不是寫死常數。

**為什麼先做這個**：這是三個階段裡風險最低、獨立性最高的一步——不動演算法，只是把資料換個存放格式。做完之後，光是「資料更新可以被 git 有意義地審閱」這一點，就已經值回票價。而且這一步做完，資料本身就變成一個可以獨立發布、被其他工具重用的資產（例如未來想做一個 CLI 版或 API 版的部件檢索，可以直接吃這份資料，不用重新解析 HTML）。

**風險/取捨**：資料檔案格式一旦定案，日後改動格式的成本會變高，值得一開始就把 schema（尤其是 `rt` 跟 `vt` 語義上的差異——一個是「展示用」，一個是「比對用單向替換」）寫清楚文件，避免後續貢獻者混用。

### 階段二：核心／UI 分層 + 建置腳本

**做什麼**：依照現有檔案裡已經存在的「客製化修改區」註解邊界，把程式拆成「核心演算法」與「UI」兩部分分別存放，再寫一個建置腳本，把資料檔 + 核心 + UI 重新組裝壓縮回一個 `部件檢索.htm`。

**為什麼是第二步而不是第一步**：這一步的正確性依賴階段一的資料格式已經穩定，先做資料抽離、驗證黃金測試案例通過之後，再動檔案分層的風險比較低。

**為什麼刻意延續「客製化修改區」邊界，而不是重新設計分層**：這個邊界不是隨意畫的——它是原作者在 [03](./03-部件檢索公開授權.md) 裡明確設計的維護契約：核心程式的更新方式是「複製貼上壓縮過的那一行」，UI 的客製化則是網站站長會直接手動修改的地方。重新發明一套分層，等於是丟掉這個已經跑了十年、被多方（包括「引得市」網站）驗證過可行的協作模式。

### 階段三：UI 現代化

**做什麼**：cookie 換成 `localStorage`（原作者自己在文中吐槽過 cookie 在本地檔案模式下會失效）、拼接字串產生 HTML 的地方換成 `<template>`/`DocumentFragment`、補上深色模式與部件鍵盤的響應式排版。

**為什麼放最後**：這一塊的優先順序最低、也最主觀——它不影響資料正確性，也不影響「單檔案可離線使用」這個核心價值。而且如果順序反過來（先做 UI 現代化），階段一、二的檔案結構調整很可能要跟著重做一次。

**明確不做的事**：不引入 React、Vue 等框架。單一檔案、零外部相依是這個專案能被「丟上任何網站」的核心原因，引入框架會需要打包工具鏈，直接違背這個約束——這不是技術債，是刻意的設計取捨，重構時要當作紅線看待。

## 怎麼知道重構沒有破壞正確性

`Eliminate()`（拆分樹比對法的實作本體）是全專案正確性風險最高的地方。[01-部件檢索辭典.md](./01-部件檢索辭典.md) 裡作者自己做過一輪「輸入 [日月]／[明]／[日日月]／[日明]」的實測比較，這四組案例被當作黃金回歸測試（完整案例與判定標準見 `AGENTS.md`）。重構過程中任何觸及資料或核心邏輯的變更，都應該先跑過這四組案例，確認「日明」是「日日月」的子集、不等於「明」這類相對關係沒有被破壞——這是原作者花了大半年時間、耗費心力才驗證出來的行為基準，不應該在重構中意外丟失。

## 現況與下一步

目前 `doc/`、`AGENTS.md` 只完成了規劃與文件化，尚未動手拆分任何程式碼。下一步建議從階段一（資料抽離）開始，理由如前述：風險最低、價值可以獨立兌現、且是後續兩個階段的前提。
