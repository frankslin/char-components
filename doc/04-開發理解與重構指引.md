本文為對 [01](./blog/01-部件檢索辭典.md)、[02](./blog/02-再談漢字的部件檢索.md)、[03](./blog/03-部件檢索公開授權.md) 三篇原作者(WFG)部落格文章，以及本專案原始 `部件檢索.htm` 原始碼(核心版本 `0.9.8.4`，2025年09月)交叉比對後的分析，目的是替後續維護、重構此專案的人（包含未來的我）提供一份「讀碼前導覽」。

**檔案已搬家**：原本放在 repo 根目錄的 `部件檢索.htm` 現已移至 [`legacy/部件檢索.htm`](../legacy/部件檢索.htm)，做為存檔保留（可離線雙擊開啟，行為不變）。本文以下內文提到的 `部件檢索.htm` 均指這個存檔版本；新實作見 [`web/`](../web/README.md)，其 `web/core.js` 是逐句對照這份存檔的核心程式碼翻譯而成。

## 1. 這是什麼專案

一個**單檔案 HTML/JavaScript 應用**，用漢字部件（偏旁、字根）組合來檢索罕用漢字，取代「猜部首、算筆畫」的傳統檢字方式。核心賣點是作者原創的**「拆分樹比對演算法」**，用來解決「漢字構形庫」「漢文博士」「剎那字引」等既有工具都存在的檢字失準問題(參見 [01](./blog/01-部件檢索辭典.md) 的實測比較表)。

專案沒有建置系統、沒有相依套件、沒有伺服器端——所有資料與邏輯都內嵌在同一個 `.htm` 檔裡，設計上就是要能「單檔案丟上任何網站」或「離線雙擊開啟即可用」（見 [03](./blog/03-部件檢索公開授權.md) 的授權說明與客製化說明）。

## 3. 現有原始碼的實際結構(對照原文用語)

用 `grep`/`python` 對 `部件檢索.htm` 做過結構性檢查，關鍵發現如下：

### 3.1 三張資料表

- **`dt`（decomposition table，拆分表）**：全部漢字的拆分資料，本質就是文中反覆提到的「漢字構形資料庫」+「字形IDSデータ」整理結果。**（2025 年資料抽離時逐字核對後訂正）** 每筆格式是 `字元 + 分支字串`，分支字串以 `@` 分隔「一字多拆」（最終完全拆分相同的替代拆法，例如 `羆@罒熊@罷灬`），以 `!` 分隔「同字異拆」（最終完全拆分不同的替代拆法，例如 `主@丶𰣀!丶王!亠土`）。索引 `1`~`10` 是保留列，內容是英文字母 `A`~`J`＋填充符號 `╳`（例如 `dt[1]='A@╳'`），供 UI 的「只搜某些擴充區」核取方塊借用同一套 `dt`/`GetIndex` 機制查找，**不是**每個漢字條目裡都會出現的「區塊旗標」；真正的擴充區篩選是在 `GetMatch()` 裡用 `GetBlock(c)` 依碼位範圍即時算出，與 `dt` 字串內容無關。
- **`rt`（variant table，異體展示表）**：與 `dt` **同索引位置**對齊的陣列，列出每個字對應的「顯示用異體」，供「解構漢字」(`\字`) 功能展示替代部件時使用，對應 [02](./blog/02-再談漢字的部件檢索.md) 裡「輔助部件鍵盤不足」的設計動機。
- **`vt`（單向映射表，即文中「異體映射」）**：`{"⻏":"邑", "⺌":"小", ...}` 形式的字典，`Variant()` 函式在比對前用它把查詢輸入與拆分表**單向摺疊**成同一個代表字，這正是 [02](./blog/02-再談漢字的部件檢索.md) 花了大篇幅推導的「ab / aB / Ab / AB 型映射」分析模型的實作——文章結論「Ab、AB 型映射有時無解」在程式碼裡直接對應：`vt` 只做單向替換，若映射方向設反了，該字就會從索引中「消失」而查不到。

### 3.2 索引方案

`GetBlock()` / `GetIndex()` 用一堆寫死的十六進位偏移量，把 Unicode 碼位轉換成 `dt`/`rt` 陣列的下標，藉此把 CJK 各擴充區(Ext-A ~ Ext-J 等)攤平進同一個陣列，並用區塊旗標做「只搜某些擴充區」的篩選（對應使用者可能只想搜基本字，不想搜罕用擴充字）。前 48 個下標是特殊保留位（符號、部件本身無法對應到正常 Unicode 排序的項目），走線性搜尋回退。

### 3.3 核心比對邏輯

- `Arrayalize(s, v)`：把使用者輸入的部件字串，依 `vt` 做異體摺疊(`v` 參數即「包容異體」勾選狀態)，轉成排序後的部件陣列。
- `Eliminate(a, s, d, v)`：**這就是「拆分樹比對演算法」的實作本體**。遞迴地在拆分樹字串 `s` 裡尋找陣列 `a` 中的每個特徵；找到就從 `a`（及走訪路徑的備份 `b`）裡摘掉，找不到就遞迴展開該部件的下一層拆分繼續找——完全對應 [02](./blog/02-再談漢字的部件檢索.md) 裡「提著檢索籃子走進漢字樹林摘葉子」的比喻。`d` 參數控制要不要跨越 `!`（即要不要考慮其他「同字異拆」分支——預設不跨越，`@` 分隔的「一字多拆」分支則一律都會嘗試）。
- `GetMatch(...)`：對 `dt` 全表跑一輪 `Eliminate`，`a.length==0` 代表全部特徵都摘光即為命中，並區分「精確命中」(`n==0`，樹上葉子也摘光)與「模糊命中」。
- `Exhaust()` / `GetTree()`：把某字的拆分樹攤開成可讀字串/HTML block，供「解構漢字」(`\字`) 功能與檢索結果的樹狀展示使用。

### 3.4 UI／核心分離

`部件檢索.htm` 檔案裡明確用註解標記出 `<!-- 客製化修改區起始/結束 -->` 區塊，這正是 [03](./blog/03-部件檢索公開授權.md) 「調整說明」第 2、3 點所描述的架構決策：客製化區放樣式、UI 事件、`ref` 跳轉連結變數；客製化區之外(往後)就是不可隨意變動、應該整行覆蓋更新的核心演算法與資料。**這個邊界是刻意設計的維護契約，重構時務必保留**，否則會破壞作者原本「核心程式更新只需複製貼上一行」的升級模型。

## 4. 對後續開發/重構的具體建議

1. **不要把 `dt`/`rt`/`vt` 當成程式碼看待。** 它們是資料，目前用「壓成一行的 JS 陣字面量」內嵌，導致單一檔案 4.3MB、`git diff` 完全不可讀。重構的第一步應該是把這三張表抽成獨立的資料檔（JSON 或自訂二進位格式），核心程式改為載入資料檔，而不是把資料寫死在 `<script>` 裡。這樣資料更新（作者提到要花大半年重新審訂拆分表）才能被 git 有意義地追蹤 diff。

2. **`GetBlock`/`GetIndex` 的魔術數字偏移量需要文件化或改成生成式。** 目前是手刻的一串 `if (c<=0x4DBF) i=c+0x1E31` 之類的偏移，任何 Unicode 版本更新（作者本人就更新過 Unicode 15.1）都要重新推導這些常數。建議寫一個小型建置腳本，從「已收錄的 Unicode 區塊列表」自動生成這個偏移對照表，而不是繼續手改。

3. **`Eliminate()` 是全專案正確性風險最高的函式，也是唯一真正實作論文級演算法的地方。** 它靠陣列的 `indexOf`/`splice`/`concat` 做原地變更與備份/還原，遞迴呼叫之間共享/複製陣列的方式不直觀。若要重構或移植到其他語言/框架，**這個函式需要優先補上單元測試**（可直接拿 [01](./blog/01-部件檢索辭典.md) 文中「輸入 [日月]／[明]／[日日月]／[日明]」四組實測案例當回歸測試，這是作者自己驗證過的黃金案例）。

4. **`vt` 的「單向映射」設計是已知有限制的，不是 bug。** [02](./blog/02-再談漢字的部件檢索.md) 的「異體映射分析模型」明確說明 `Ab` 型與部分 `AB` 型映射無解（含有該部件的字會查不到）。日後若要擴充 `vt` 表、新增異體對應關係，**必須先用文中的 ab/aB/Ab/AB 分類法判斷映射方向**，否則會重新製造出作者當年耗費數月才抓出的「檢索結果偶有缺漏」問題。

5. **保留「客製化修改區」邊界。** 任何重構若要拆分 UI 與核心邏輯到不同檔案，應延續 `<!-- 客製化修改區起始/結束 -->` 這條既有邊界作為模組切分依據，而不是重新發明一套分層。

6. **`0.9.8.4` 版本已遠遠超出三篇原文描述的「0.90 最終測試版」**，代表 2015 年之後作者仍持續維護到 2025 年。後續的重要公告已於 2026/07 逐字備存於 [doc/blog/](./blog/README.md)（04～13，部落格正文 + 留言選錄，見 5.6 節的對照表），其中兩篇直接對應本 repo 版本：
   - 2023/10/11：「漢字使用環境的建置 —— 部件檢索重要更新暨 Unicode 15.1 全宋體更新」（`LICENSE` 檔已提及）——備存於 [doc/blog/12](./blog/12-漢字使用環境的建置——部件檢索重要更新暨Unicode15.1全宋體更新.md)

## 5. 字型依賴：「全宋體」

本專案的正確顯示，強依賴一套字型——「**全宋體**」。這件事原文三篇(01/02/03)完全沒提到(當時作者推薦的是「花園明朝體」)，是交叉比對 2023/10/11、2025/09/12 兩篇後續公告文章後才確認的關鍵環境依賴，重要性等同於「客製化修改區邊界」「`vt` 單向映射限制」，一併記錄於此。

### 5.1 「全宋體」是什麼

「全宋體」是作者 WFG **個人整理、編譯**的大型宋體字型專案，**不是**「台灣全字庫正宋體」（後者是官方 CNS11643 標準字型，兩者規模、性質都不同；「全字庫」收錄的官方標準字只是「全宋體」眾多覆蓋範圍之一）。依 2025/09/12 公告，「全宋體」目前(對應 Unicode 17.0)總計收錄 **202,664 個漢字**：

- 102,376 字是 Unicode 已正式編碼的漢字(基本字區 + Ext A~J)。
- 另外 **100,288 字是 Unicode 尚未編碼的「補充字」**，作者用私有造字區(PUA)碼位暫存，待該字被 Unicode 正式收錄後再「遷碼」到官方碼位、註銷舊的私有碼位。

字型本身的玻璃字形有部分是整合自其他既有字型專案再優化(公告鳴謝「原字型製作單位」)，也有大量作者與協作者(jcz777、suns99 等)自行新造的字形，因此更準確地說是「編譯 + 校訂 + 新造」的綜合成果，而非從零獨立設計。

### 5.2 為什麼不裝就會顯示錯亂

`部件檢索.htm` 的 CSS 直接寫死了「全宋體」的內部字體家族名：

```css
font-family: Calibri, FSung-2, FSung-m, FSung-3, FSung-F, FSung-X, FSung-1, MingLiU, ...
```

`FSung-1`～`FSung-X` 就是「全宋體.zip」拆分成多個檔案安裝時各自的字體家族名(20 萬字塞不進單一字型檔的格式限制)。這代表 `部件檢索.htm` 從樣式表層級就是**跟「全宋體」綁定發行**的，不是泛用的「裝個中文字型就好」。

風險不只是缺字：

- **沒裝任何相容字型**：`dt` 表裡那 10 萬個私有碼位補充字會顯示成「豆腐塊」(空心方框)，這是最溫和的失敗模式。
- **裝了別的字型，剛好也用了同樣的 PUA 碼位**(例如改裝「全字庫正宋體」)：PUA 碼位沒有全球共通標準，同一個碼位在不同字型裡可能對應到完全不同的字。這種情況下**不會缺字，而是會顯示成另一個不相干的字**，比缺字更隱蔽——使用者查到的結果可能是錯的，卻毫無察覺。

### 5.3 對重構的影響

- 若日後把 `dt`/`rt`/`vt` 資料抽成獨立檔案（見「4. 對後續開發/重構的具體建議」第 1 點），資料 schema 文件裡應該明確標註「哪些碼位落在 PUA、依賴哪個版本的全宋體才能正確顯示」，否則資料一離開原本的 HTML/CSS 綁定環境，這個隱性依賴很容易被忽略。
- 字型與 `dt` 拆分表的版本必須互相對應，不能混搭不同時期的字型與資料表——這套字型仍在持續遷碼更新(隨 Unicode 新版本發布)。

### 5.4 字型檔案已存入本 repo

字型檔案已下載存放於 `deps/fonts/`，來源為 2025/09/12 公告「[漢字使用環境的建置——Unicode 17 全宋體更新](https://fgwang.blogspot.com/2025/09/unicode-17.html)」(對應本 repo `部件檢索.htm` 核心版本 `0.9.8.4`)提供的下載連結，屬同一批次、版本互相對應：

| `FSung-p.ttf` | 同批次隨附的第七個字型檔，目前 `部件檢索.htm` 的 CSS 未見引用；推測是尚未在這個核心版本用到的補充分卷，用途待後續版本或原文確認 |

### 5.5 Webfont 版本：`webfonts/wfg-fsung/`

為了讓「全宋體」也能當 web 字型使用(而不是只能離線安裝到系統)，`webfonts/wfg-fsung/` 提供了一份切片版本，做法參考鄰近的 `jigmo-webfonts`/`free-fonts` 專案慣例(256 碼位一個 WOFF2 分片 + `unicode-range` CSS，讓瀏覽器只下載頁面實際用到的區塊)。細節見 `webfonts/wfg-fsung/README.md`；建置腳本 `webfonts/build-wfg-fsung.py` 可從 `deps/fonts/FSung-*.ttf` 重新產生。

幾個值得記錄的發現：

- `FSung-p.ttf` 的 cmap 與 `FSung-m.ttf` **完全重複**(皆為 40,582 碼位)，且原始 `部件檢索.htm` 的 CSS 從未引用 `FSung-p`，因此這個 webfont package **排除了 `FSung-p`**，只切了另外六個檔案。
- 六個檔案彼此幾乎不重疊，唯一的例外是 U+0000–00FF(Basic Latin)這個分片：`FSung-m` 在此區間的碼位是其餘各檔同分片內容的**超集**，因此該分片整塊歸給 `FSung-m`，不需要真正的跨檔合併(build script 的 `assign_chunks()` 會做這個「超集檢查」，若未來字型更新後出現真正無法用超集解決的跨檔分片，腳本會直接報錯中止，而不是默默漏字)。
- 這個 package **不可與 `free-fonts` 裡其他 `@free-fonts/*` 套件混為一談**：全宋體的授權是「無條件分享給非商業用途，禁止任何商業營利」，比 `free-fonts` 現有字型的 MIT/OFL/CC0(允許商業散布)嚴格得多。因此 npm 套件名稱訂為獨立的 `wfg-fsung-webfonts`，**不掛在 `@free-fonts` scope 下，也不放進 `free-fonts` 的 README/index.html 展示頁**，`package.json` 用自訂的 `LicenseRef-WFG-NonCommercial` 標示，避免使用者誤以為這套字型跟其他 `@free-fonts/*` 套件一樣可以商業使用。發布到 npm 由使用者本人處理。
- package 目錄本身(`webfonts/wfg-fsung/`)的 `README.md`/`LICENSE` 刻意寫成**不依賴目錄外任何東西**(不引用 repo 裡其他路徑)，因為這個目錄會被整包獨立發布到 npm，安裝後只剩 `package.json` 列出的檔案(`wfg-fsung.css`、`fonts/`、`README.md`、`LICENSE`)，任何指向 repo 內其他檔案的相對連結到了使用者手上都會是死連結。

**授權提醒**：「全宋體」的分享聲明與 [03-部件檢索公開授權.md](./blog/03-部件檢索公開授權.md) 的精神一致——無條件開放給學術研究、教育、個人閱讀使用，**但禁止任何商業營利用途**。這批字型檔屬於原作者的另一份獨立授權範圍(字型本身，非「部件檢索」程式)，本 repo 存放時應保留來源出處，且沿用同樣的非商業限制。

**倉儲體積提醒**：`deps/fonts/` 七個 `.ttf` 加總約 200MB，屬二進位大檔。目前直接存進 git 歷史，日後若倉儲體積成為問題，可考慮改用 Git LFS 或改為建置時下載的外部資產，而不是一開始就用 LFS 增加操作複雜度。

**PUA 碼位分配（實測 `dt.jsonl` 205,207 個字頭得出，非作者文件）**：

- 補充字**連續無空洞**地佔用 U+F0200–U+FFFFD（Plane 15，65,022 字）和 U+100000–U+1090EB（Plane 16，37,100 字）。Plane 15 開頭的 U+F0000–U+F01FF（512 個碼位）空置，字型 cmap 同樣沒有，推測為保留區。
- **BMP PUA（U+E000–F8FF）完全不用**：`dt` 全表和七個字型檔的 cmap 都是零。唯一例外是 `rt.jsonl` 裡出現一個 U+E54E，且它不在任何 FSung 字型的 cmap 裡，疑似資料殘留，待查。
- 字型檔按 Unicode 平面切分（部落格留言區證實）：`FSung-m`=BMP、`FSung-1/2/3`=Plane 1/2/3、`FSung-F`=Plane 15、`FSung-X`=Plane 16。
- 「零空洞」意味著：遷碼騰出的舊碼位會被**回收回填**（2018 公告原文「空下的碼位有可能被回收利用」）。因此 **PUA 碼位跨版本完全不穩定**——同一碼位在不同版本的全宋體裡可能是不同的字，比「缺字」更危險。

**遷碼機制與唯一的官方交換資料**：每逢 Unicode 新版收錄了某批補充字，作者隨新字型發布一份「補充字對Ｘ區字遷碼對照表」（G、H、I、J 區各有一份 zip，掛在對應公告的 Google Drive 連結；J 區那次遷了 1,870 字）。這是作者提供的**唯一**碼位對照資料——沒有逐字標註「出自哪部字書第幾頁」的屬性總表，也沒有完整的補充字清單可下載（`dt` 拆分表本身就是最接近總清單的東西）。

**對本 repo 長期可用性的結論**：

1. 資料表(`dt`/`rt`)、字型檔、遷碼對照表三者必須**鎖同一版本**使用（本 repo 為 Unicode 17.0 / 0.9.8.4 批次），這點 5.3 已述；本節補充的是「為什麼」——碼位會回收回填。
2. 跨版本升級資料時，**必須先取得對應的遷碼對照表**把舊文本遷碼，否則舊 PUA 碼位可能已被回填成別的字。歷次對照表只掛在 Google Drive，隨時可能失效，**建議下載存入 `deps/`**（目前尚未存）。
3. 與外部系統交換含補充字的文本，目前**沒有**穩定句柄；可行的替代是交換 IDS 拆分式（`dt` 資料可生成），或等該字被 Unicode 正式收錄後用正式碼位。任何把 PUA 碼位寫進持久化資料的設計，都必須同時記錄「全宋體版本號」才有還原可能。
