# data/ — 抽離自 `部件檢索.htm` 的原始資料表

這是重構 Phase 1 的產出（見 `AGENTS.md`、`doc/05-重構計畫.md`）：把內嵌在 `部件檢索.htm` 單一 `<script>` 裡、壓成一行的 `dt`/`rt`/`vt` 三個 JS 陣列字面量，原封不動地抽成獨立檔案，方便 `git diff` 追蹤資料變動。**目前只做抽離，核心演算法與 UI 都還沒動**，`部件檢索.htm` 本身也還沒有被改成從這裡讀資料——這一步純粹是把資料搬出來、驗證過完全等價而已。

用 `python3 data/extract.py` 可重新從 `部件檢索.htm` 產生這三個檔案；已用逐項比對確認抽離結果與原始 JS 陣列**完全一致**（見該腳本的 round-trip 驗證）。

## 三張表分別是什麼

### `dt.jsonl` — 拆分表（decomposition table）

一行一筆，**第 N 行（0-indexed）對應原本 `dt[N]`**，順序即索引，不要重新排序這個檔案的行——`GetIndex()`/`GetBlock()` 是靠位置換算下標，不是靠排序找字。

每筆格式是：

```
<字元><分支字串>
```

- `<字元>`：該筆的漢字本身（超過 BMP 的字用 UTF-16 surrogate pair，Python/JS 字串處理上沒問題，但用位元組層級工具處理要小心不要切斷 surrogate pair）。
- `<分支字串>`：以 `@` 或 `!` 開頭，記錄這個字的拆分資訊：
  - `@` 分隔「**一字多拆**」：多種拆法，但每種拆法最終「完全拆分」的結果都相同。例如 `羆@罒熊@罷灬` 表示「羆」可拆成「罒熊」或「罷灬」，兩者最終完全拆分相同（見 [doc/02](../doc/02-再談漢字的部件檢索.md)）。
  - `!` 分隔「**同字異拆**」：多種拆法，但每種拆法最終完全拆分的結果不同。例如 `主@丶𰣀!丶王!亠土` 表示「主」有三種拆法，其中「丶王」與「亠土」是同字異拆的兩個不同結果。
  - 分支字串裡的每個字元(除了 `@`/`!` 本身)，代表這個字的一個部件；該部件若還能再拆，要另外去 `dt` 表裡查它自己的條目遞迴展開——**拆分內容不會展開巢狀寫在同一個字串裡**，遞迴發生在演算法（`Eliminate()`）裡，不是資料格式裡。
- **`dt[1]`~`dt[10]` 是保留列**，內容是 `A@╳`、`B@╳`...`J@╳`：這不是漢字資料，是 UI 用來借同一套 `GetIndex()` 機制查「Unicode 擴充區篩選核取方塊」用的佔位項，跟其他索引位置的漢字條目語意完全不同。實際的「只搜某些擴充區」篩選邏輯是 `GetMatch()` 用 `GetBlock(c)` 依碼位範圍即時算出來的，**跟 `dt` 字串內容無關**——早期版本的 `doc/04` 曾誤寫成「`dt` 每筆格式包含區塊旗標」，已訂正。

### `rt.jsonl` — 展示異體表（variant-for-display table）

一行一筆，**第 N 行對應原本 `rt[N]`，與 `dt.jsonl` 同一組索引、逐行對齊**（`rt[N]` 是 `dt[N]` 那個字的「顯示用異體」清單）。用於「解構漢字」(`\字`) 功能展示替代部件時參考，跟 `vt.json` 的「比對用單向映射」是完全不同的兩件事，不要混用（`rt` 只影響顯示建議，`vt` 才會實際改變比對結果）。

### `vt.json` — 單向異體映射表（variant mapping for matching）

一個 JSON 物件（852 筆），`{"部件": "映射目標部件", ...}`，保留原始宣告順序。比對前會用這張表把使用者輸入與拆分表裡的部件單向摺疊成同一個代表字（「包容異體」選項）。**這張表只能單向替換**——擴充前務必讀 [doc/02](../doc/02-再談漢字的部件檢索.md) 的 ab/aB/Ab/AB 型映射分析，`Ab`、部分 `AB` 型映射目前無解，方向設反會讓某些字從索引裡「消失」查不到。

## 目前的限制 / 尚未做的事

- 這三個檔案目前是**唯讀的抽離結果**，`部件檢索.htm` 本身還是用自己內嵌的舊資料在跑，兩邊尚未接起來。Phase 2（核心/UI 分層 + 建置腳本）會處理「怎麼讓 `部件檢索.htm` 改成從這裡讀資料再組裝回單一檔案」。
- `GetBlock()`/`GetIndex()` 那組手刻的 Unicode 區塊偏移量常數，還沒有跟著資料一起抽出來或生成式化（仍是 `AGENTS.md` Phase 1 待辦事項之一）。
- 檔案大小：`dt.jsonl`/`rt.jsonl` 各約 20 萬行，`git diff` 應該已經比原本壓成一行的版本好讀很多，但尚未實測真實資料更新時的 diff 品質（作者提到還有大半年的拆分表校訂工作）。
