# AGENTS.md

指引在本 repo 工作的 AI coding agent。人類可讀的完整規劃見 `doc/05-重構計畫.md`；本檔只放 agent 執行時必須遵守的規則與任務拆解。

## 專案現狀

- 唯一的產出物是 `部件檢索.htm`：單一 HTML 檔，內含全部漢字拆分資料(`dt`/`rt`/`vt` 三個巨型陣列)與比對演算法(`Eliminate`/`GetMatch`/`Arrayalize`/`Exhaust`/`GetTree`)，無建置系統、無相依套件。
- 背景與演算法原理見 `doc/01`~`doc/04`。**動手改演算法前必讀 `doc/04-開發理解與重構指引.md`**，尤其是「拆分樹比對法」與「異體映射」章節——這兩塊最容易改錯。
- `LICENSE` 是原作者的公開授權條款，禁止商業使用；任何衍生產出物需保留來源標示。

## 不可破壞的約束(Definition of Done 的一部分)

1. **最終產出必須仍是可離線雙擊開啟、可整檔丟上任何網站的單一 HTML 檔。** 拆源碼是為了讓開發可維護，不是要把產品變成需要建置環境才能跑的東西。
2. **保留「客製化修改區」邊界。** 現有檔案用 `<!-- 客製化修改區起始/結束 -->` 註解劃出 UI/樣式可自由改動的範圍，核心演算法在此範圍外。任何拆檔方案都要延續這條邊界，不要重新發明分層。
3. **`vt`(異體映射表)只能單向替換。** 新增或修改映射前，先用 `doc/02` 的 ab/aB/Ab/AB 分類法判斷方向是否安全；`Ab`、部分 `AB` 型映射目前無解，硬加會導致某些字查不到。
4. **`Eliminate()` 是全專案正確性風險最高的函式。** 任何觸碰它的重構，必須先用下面的黃金測試案例跑過一遍，確認結果不變。

## 黃金回歸測試案例(來自 `doc/01`，作者本人驗證過)

以查詢輸入 → 預期命中數/字集，至少要能重現以下四組：

| 輸入 | 完拆比對法結果 |
|---|---|
| `日月` | 99 字(與剎那字引一致) |
| `明` | 99 字 |
| `日日月` | 7 字：𣇵 𣊧 𣊿 𣋂 𣋐 𣎱 𦣖 |
| `日明` | 7 字：𣇵 𣊧 𣊿 𣋂 𣋐 𣎱 𦣖(與「日日月」相同) |

注意：現行程式跑的是「拆分樹比對法」而非文中的「完拆比對法」，數字可能因拆分表版本更新而略有出入(現在的 `dt` 已是 Unicode 17.0 版，非 2015 年的舊表)，但**同一批輸入彼此之間的相對關係**(例如「日明」應該是「日日月」的子集，不應該等於「明」)必須維持，這才是回歸測試真正要保護的不變量，而非死板的絕對數字。

## 任務拆解與優先順序

### Phase 1 — 資料抽離(建議優先做，風險最低、獨立性最高)

- 把內嵌在 `<script>` 裡的 `dt`/`rt`/`vt` 陣列字面量抽成獨立資料檔(建議每字一行的 JSON Lines 或等價格式，方便 git diff)。
- 把 `GetBlock`/`GetIndex` 裡手刻的 Unicode 區塊偏移量，改成從「已收錄 Unicode 區塊列表」生成，而不是寫死的十六進位常數。
- 產出：`data/dt.jsonl`、`data/rt.jsonl`、`data/vt.json` + 一份 schema 說明(哪個欄位是什麼語義，尤其要區分 `rt` 的「展示用異體」跟 `vt` 的「比對用單向映射」，這兩者語義不同，不要混用)。
- 完成標準：舊版 `部件檢索.htm` 與「用新資料檔 + 舊核心邏輯重新組裝出的 htm」在黃金測試案例上結果一致。

### Phase 2 — 核心/UI 分層 + 建置腳本

- 依現有「客製化修改區」邊界，把檔案拆成 `core/`(演算法，純函式、無 DOM 依賴) 與 `ui/`(現有客製化修改區內容)。
- 寫一個建置腳本(Node 即可，不需要複雜工具鏈)，把 `data/` + `core/` + `ui/` 重新組裝壓縮回單一 `部件檢索.htm`，作為 `dist/` 產物。
- 完成標準：`dist/部件檢索.htm` 與原檔行為一致，且可離線雙擊開啟。

### Phase 3 — UI 現代化(優先度最低，做完前兩步再排)

- cookie → `localStorage`。
- 字串拼接生成 HTML(`MakeBlock` 等) → `<template>`/`DocumentFragment`。
- 補 `prefers-color-scheme` 深色模式、部件鍵盤的響應式排版。
- **不要引入 React/Vue 等框架**——單檔案零依賴是這個專案的核心約束，不是技術債。

## 給 agent 的具體提醒

- 改動 `dt`/`rt`/`vt` 資料前，先確認是資料錯誤還是演算法錯誤——多數「查不到某字」的回報，根源在 `vt` 映射方向錯誤(見上面約束 3)，不要急著去改 `Eliminate()`。
- 這個 repo 沒有 CI、沒有測試框架。新增任何自動化測試時，優先用 Node 內建的 `node:test` + `assert`，不要為了跑測試而引入額外相依套件，這與「單檔案零依賴」的專案精神一致。
- 提交訊息、程式碼註解一律使用繁體中文或英文皆可，但不要新增簡體中文內容(維持與現有原文/文件一致的用字)。
