# web/data/ — 抽離自 `legacy/部件檢索.htm` 的原始資料表

這是重構 Phase 1 的產出（見 `AGENTS.md`、`doc/05-重構計畫.md`）：把內嵌在 `legacy/部件檢索.htm` 單一 `<script>` 裡、壓成一行的 `dt`/`rt`/`vt`/`kt` 幾個 JS 字面量，原封不動地抽成獨立檔案，方便 `git diff` 追蹤資料變動。`web/` 底下的新實作會讀取這裡的檔案；`legacy/部件檢索.htm` 本身仍是原封不動的封存版本，維持用自己內嵌的舊資料在跑。

後來另外加了兩份**不是**從 legacy 抽出來的資料：`bt.json`（部首筆畫鍵盤表）與 `moe/`（教育部異體字字典字號對照表），各自的來源見下方小節。

用 `python3 web/data/extract.py` 可重新從 `legacy/部件檢索.htm` 產生這些檔案（只產生 `dt`/`rt`/`vt`/`kt`，不碰 `bt.json`、`moe/`）；已用逐項比對確認抽離結果與原始 JS 陣列**完全一致**（見該腳本的 round-trip 驗證），**唯一的例外**是 `extract.py` 會自動套用 `align_pua_rows()`，更正 legacy 裡 `U+F7121–F712F` 這 15 列的錯序（見下方 `dt.jsonl` 小節；此步冪等，重跑不必手動補回）。

## 各檔案分別是什麼

### `dt.jsonl` — 拆分表（decomposition table）

一行一筆，**第 N 行（0-indexed）對應原本 `dt[N]`**，順序即索引，不要重新排序這個檔案的行——`GetIndex()`/`GetBlock()` 是靠位置換算下標，不是靠排序找字。（唯一刻意重排過的是 `U+F7121–F712F` 這 15 列，那是為了**修正** legacy 的錯序，見本節末。）

每筆格式是：

```
<字元><分支字串>
```

- `<字元>`：該筆的漢字本身（超過 BMP 的字用 UTF-16 surrogate pair，Python/JS 字串處理上沒問題，但用位元組層級工具處理要小心不要切斷 surrogate pair）。
- `<分支字串>`：以 `@` 或 `!` 開頭，記錄這個字的拆分資訊：
  - `@` 分隔「**一字多拆**」：多種拆法，但每種拆法最終「完全拆分」的結果都相同。例如 `羆@罒熊@罷灬` 表示「羆」可拆成「罒熊」或「罷灬」，兩者最終完全拆分相同（見 [doc/blog/02](../../doc/blog/02-再談漢字的部件檢索.md)）。
  - `!` 分隔「**同字異拆**」：多種拆法，但每種拆法最終完全拆分的結果不同。例如 `主@丶𰣀!丶王!亠土` 表示「主」有三種拆法，其中「丶王」與「亠土」是同字異拆的兩個不同結果。
  - 分支字串裡的每個字元(除了 `@`/`!` 本身)，代表這個字的一個部件；該部件若還能再拆，要另外去 `dt` 表裡查它自己的條目遞迴展開——**拆分內容不會展開巢狀寫在同一個字串裡**，遞迴發生在演算法（`eliminate()`）裡，不是資料格式裡。
- **空字串列**（`""`）代表「這個索引位置沒有字元」：是 `extract.py` 的 `blank_noncharacter_rows()` 抹掉的無字形佔位列，見下方小節。`core.js` 讀到空列會直接跳過（`getMatch` 不列入結果、`getIndex` 相關路徑視為不可分解）。
- **`dt[1]`~`dt[10]` 是保留列**，內容是 `A@╳`、`B@╳`...`J@╳`：這不是漢字資料，是 UI 用來借同一套 `GetIndex()` 機制查「Unicode 擴充區篩選核取方塊」用的佔位項，跟其他索引位置的漢字條目語意完全不同。實際的「只搜某些擴充區」篩選邏輯是 `GetMatch()` 用 `GetBlock(c)` 依碼位範圍即時算出來的，**跟 `dt` 字串內容無關**。

#### `U+F7121–F712F` 錯序修正（`align_pua_rows`）

**`U+F7121` 這個碼位，對應的字應該是「糹殿」（糹+殿）。** 在「全宋體」的字形檔裡，U+F7121 畫的就是「糹殿」；但 legacy 的 `dt`/`rt` 資料裡，「糹殿」的條目卻被放在這 15 列視窗的**最後一列**（算術位置 U+F712F），其餘 14 字各往後移一格——也就是資料的列序比字型的碼位晚了一格，兩者對不上。

這造成的症狀：`getIndex()`（`core.js`）是純算術 `id = 碼位 - 0xD6F55`，用它查這 15 個補充字時會取到**隔壁列**。`getMatch()`（顯示結果時讀內嵌字碼）看起來正常，但 `getTree()`／字詳情（走 `getIndex()`）會畫出「後一個碼位」的字——例如點「糹殿」看拆分，畫出來的卻是「氵吂朋凡」的拆分。

`extract.py` 的 `align_pua_rows()` 已把這 15 列按內嵌碼位升序重排（等同把最後一列 rotate 回開頭），讓 **列序 == 內嵌碼位 == 全宋體字形** 三者一致（U+F7121 的列 = 「糹殿」）。`rt.jsonl` 同列套用相同排列以維持逐行對齊。**只更正列序，不改任何 char↔拆分 綁定**；且是**對齊已發布的全宋體碼位**，使用者不需要更換字型。

#### 抹除無字形的非字元佔位列（`blank_noncharacter_rows`）

`dt` 的算術區塊（`getIndex()`）會把每個 Unicode 區塊補齊到區塊邊界，於是納入一些**未指派、全宋體也畫不出字形**的碼位；PUA 補充字塊開頭（U+F0200–F021F）也有一段保留槽。legacy 把這些列填成 `字@╳`（無拆分、也沒被任何字當部件），但 `getMatch()` 會照樣把它們當結果丟出——例如**搜 U+3347A 會回 6 個一模一樣的空框**（3347A 本尊＋5 個重複佔位列，因為 Ext-J 尾端 padding 都重複塞了 3347A）。

`extract.py` 的 `blank_noncharacter_rows()` 把這些列（碼位落在 `NONCHAR_RANGES`、`i>=49`）抹成空字串 `""`；搭配 `core.js` 的 `if (!dt[i]) continue;`，非字元就不再被搜出（`搜 3347A → 0 個結果`），真字則完全不受影響。`NONCHAR_RANGES` 由 `deps/fonts/` 的全宋體 cmap 與 `dt` 逐碼位比對得出（**版本綁定**：字型或資料改版時要照 `extract.py` 註解裡那行重算）。目前共抹除 68 列（63 個碼位，其中 3347A 佔 6 列）。

### `rt.jsonl` — 展示異體表（variant-for-display table）

一行一筆，**第 N 行對應原本 `rt[N]`，與 `dt.jsonl` 同一組索引、逐行對齊**（`rt[N]` 是 `dt[N]` 那個字的「顯示用異體」清單）。用於「解構漢字」(`\字`) 功能展示替代部件時參考，跟 `vt.json` 的「比對用單向映射」是完全不同的兩件事，不要混用（`rt` 只影響顯示建議，`vt` 才會實際改變比對結果）。

### `vt.json` — 單向異體映射表（variant mapping for matching）

一個 JSON 物件（853 筆），`{"部件": "映射目標部件", ...}`，保留原始宣告順序。比對前會用這張表把使用者輸入與拆分表裡的部件單向摺疊成同一個代表字（「包容異體」選項）。**這張表只能單向替換**——擴充前務必讀 [doc/blog/02](../../doc/blog/02-再談漢字的部件檢索.md) 的 ab/aB/Ab/AB 型映射分析，`Ab`、部分 `AB` 型映射目前無解，方向設反會讓某些字從索引裡「消失」查不到。

本專案在 legacy 原始 852 筆之外的人工增補（`extract.py` 重新產生時**不會**包含這些，重跑後需手動補回）：

- `㇣→〇`：筆畫「圈」(U+31E3，拆分表用字) → 中文數字零「〇」(U+3007，輸入法可打)。兩者皆為終端部件，屬 ab 型，方向安全；沿用「拆分表內部用字 → 使用者打得出的正字」慣例（同 `⻏→邑`、`𠆢→人`）。

### `kt.json` — 部件鍵盤分類表（component keypad categories）

一個 JSON 陣列（22 筆），供側邊選字面板使用。每筆格式是：

```
<圖示>,<分類名稱>:<第一列字元>,<第二列字元>,...
```

冒號前是「圖示,分類名稱」（原始 UI 裡圖示與名稱之間用 `<br>` 換行顯示，這裡沿用同一個逗號分隔）；冒號後每個逗號分隔的區段是鍵盤上的一「列」，列裡的每個字元各自是一顆單獨的按鍵（不是一串詞，是一組相關部件並排展示）。例如：

```
"𑇆,筆畫:一丨丶,㇒丿㇀,㇈亅乚,..."
```

代表分類圖示「𑇆」、名稱「筆畫」，第一列是「一」「丨」「丶」三顆按鍵，第二列是「㇒」「丿」「㇀」三顆按鍵，依此類推。`web/keypad.js` 的 `parseKeypad()` 會把這個字串格式轉成 `{ icon, name, rows }` 物件陣列。

### `bt.json` — 部首筆畫鍵盤表（Kangxi radicals by stroke count）

**不是**從 `legacy/部件檢索.htm` 抽出來的**（`extract.py` 不會產生也不會覆寫這個檔）**，是康熙 214 部首依筆畫數分組的標準索引表，供側邊鍵盤的第二種排列「依部首筆畫」使用。加這張表的原因：原本只有 `kt.json` 那種「部件分類」鍵盤，使用者回報想找的部件不知道歸在哪一類就找不到（回報範例是「厷」）；部首表是大家從紙本字典就熟悉的索引方式，畫數是客觀屬性，不必猜分類。

一個 JSON 物件，兩個鍵：

```json
{
  "kangxi":      ["<畫數>:<部首1>,<部首2>,...", ...],   // 17 筆，1~17 畫各一筆
  "simplified":  ["<畫數>:<簡化形><部首編號>,...", ...], // 8 筆，只有有簡化形的畫數
  "strokeForms": ["<畫數>:<筆形部件們>", ...]            // 2 筆(1 畫、2 畫)
}
```

**`kangxi`**：冒號後每個逗號分隔的區段是**一個部首**（注意跟 `kt.json` 不同，那裡一個區段是「一列按鍵」），區段裡第一個字元是正形、其餘是附形，例如 `人亻`、`水氵氺`、`心忄㣺`。**部首編號（1~214）不存在資料裡**，由陣列順序推算。

**`simplified`**：簡體字形的部首（`马187` = 部首 187 的簡化形「马」）。關鍵是**它們歸在自己的畫數底下**——`马` 在 3 畫、`馬` 在 10 畫，兩邊都要找得到；掛在繁體正形旁邊當附形的話，簡體使用者在 3 畫組裡就是找不到「马」。編號寫在資料裡而不是靠順序，因為簡化形跟康熙順序無關。共 26 筆：

| 畫數 | 簡化部首 |
|---|---|
| 2 | 讠 |
| 3 | 么 纟 门 飞 饣 马 |
| 4 | 户 见 贝 车 长 韦 风 |
| 5 | 钅 鸟 龙 |
| 6 | 页 齐 |
| 7 | 卤 麦 龟 |
| 8 | 鱼 黾 齿 |
| 11 | 黄 |

**`strokeForms`**：`kt.json`「筆畫」分類那批**筆形部件**（㇒ ㇄ 𠃊 乛 ⺄ …），冒號後字元直接相連（每個字元各自是一顆按鍵，不用逗號分隔）。它們不是部首，但在拆分表裡是最基本的葉節點、用量極大（㇒ 有 1273 個字用到），**只列 214 部首的話 1 畫組就只剩「一丨丶丿乙亅」六顆**，等於把最常用的一批部件擋在門外。原分類 39 個字裡有 5 個（一丨丶丿亅）本來就是康熙部首，不重複列，其餘 34 個分成 1 畫 24 個、2 畫 10 個。畫數來源：

- Unihan `kTotalStrokes`（一般字，如 𠃊 𠃍 乀 乂 巜）。
- **CJK Strokes 區（U+31C0–U+31E3）依定義都是單筆**（㇒ ㇀ ㇈ ㇄ …），Unihan 沒有這些碼位的資料，但這個區收的就是「一個筆畫」。
- 剩下 7 個沒有任何權威資料（5 個 PUA 補充字 ＋ ⺀ ＋ ⺄）：用 `dt` 反推——找「第一分支裡除了它以外筆畫都已知、且該字有 `kTotalStrokes`」的字，用總筆畫減去其他部件，取眾數。結果 ⺄=1（56/63）、⺀=2（34/38）、󰊸=2（21/28）、󰕐=2（32/37）、󰊅=2（9/11）；󰎀（4/5）與 󰋌（樣本只有 5、3:2 分歧）證據最薄，都取 2——󰋌 可以用「卍（4 畫）= 󰋌+𠃍+一」這個人工核得動的樣本佐證。要重算就重跑這個方法（見 git log 裡這次的推導腳本說明）。

`web/keypad.js` 的 `parseRadicalPad()` 把三個鍵合併成依畫數排序的 `{ strokes, radicals: [{ no, forms }], simplified: [{ no, form, trad }], strokeForms: [字…] }`。

- `kangxi` 的內容整理自香港中文大學《漢語多功能字庫》的[漢字部首索引](https://humanum.arts.cuhk.edu.hk/Lexis/lexi-mf/radical.php)（214 部首的畫數分組與附形），是通行的標準部首表，不是本專案自行推算的畫數——`dt` 的拆分資料**沒有**筆畫數欄位，想從拆分遞迴推算畫數實測與 Unihan `kTotalStrokes` 只有約 75% 吻合，會把字放進錯的畫數分組，所以不走那條路。
- `simplified` 的清單與畫數同樣不是手數的，是從 **Unihan** 機械推導：`kRSUnicode` 標成 `N'.0`（部首 N 的簡化形、殘餘筆畫 0）的字，加上 214 部首各形的 `kSimplifiedVariant`／反向 `kTraditionalVariant`；畫數一律取 `kTotalStrokes`。人工只做了排除：相容表意文字碼位（U+2FA18 的重複「黾」）與 `kRSUnicode` 標兩個引號 `N''.0` 的日式略字／罕見異體（鼡、𲋄、𬺞），這些不是簡體中文的標準字形。要重跑就照這個規則對 `Unihan_IRGSources.txt` / `Unihan_Variants.txt` 重掃一次。
- 26 個簡化部首在 `dt` 裡的使用次數全部 > 0（最少的「黄」20 次），沒有死鍵；`vt` 也都有對應的單向映射（讠→言、马→馬…），所以預設開著「包容異體」時，繁簡兩邊的鍵查出來的結果會合在一起。
- **刻意刪掉 5 個附形**：`𡯁`（部首 43 的正形）、`⺔`、`⺵`、`⺴`、`⺿`。這 5 個字形在 `dt` 的拆分內容裡一次都沒被用到（連 `vt` 折疊後也沒有），留著就是查不出東西的死鍵；每個部首都至少還留有一個 `dt` 用得到的形（如 43 剩 `尢兀尣𡯂`、140 剩 `艸`）。要重新驗證這件事，可以掃 `dt.jsonl` 的分支字串統計每個字形被當部件用到的次數。
- 這張表跟拆分資料版本無關（部首表是固定的），`dt`/`rt` 改版時不需要跟著重產。

### `moe/` — 字頭 ↔ 教育部異體字字典字號對照表（分片）

**不是**從 `legacy/部件檢索.htm` 抽出來的，來源是 WFG 另行發布的《[教育部異體字索引字典](https://fgwang.blogspot.com/2021/12/blog-post_29.html)》離線字典（mdx，本表基於第七版、CreationDate 2025-9-24）。收錄《教育部異體字字典》全部 96,933 個字頭（正字 29,920＋異體 74,381＋附字 2,002，同字多重身份合併計）的字號對照，其中 41,521 個是全宋體的 PUA 補充字。`web/` 的字詳情面板用它把「外部字典查不到」的 PUA 字直鏈到教育部官網對應頁面，已編碼字則從「以字搜尋」升級為字號直鏈。

全表數 MB 但單次查詢只需其中一片，故分片存放，只產出非空片，粒度與 `web/data.js` 讀取端必須一致：

- **正向（字 → 字號）**：按 **4096 碼位一片**（`碼位 >> 12`，即 `extract_moe.py` 的 `SHARD_SHIFT` = `web/data.js` 的 `MOE_SHARD_SHIFT`），檔名是片首碼位的六位十六進制（如 `004000.jsonl` 收 U+4000–U+4FFF），共 52 片、中位數 47KB、最大 113KB。
- **反向（字號 → 字）**：供「檢索字號」功能反查用，按字號千位一片，檔名 `code-<字母><千位兩位數>.jsonl`（如 `code-a02.jsonl` 收 A02000–A02999 的全部身份，含 `-NNN` 後綴），共 32 片。一行 `["<字號>", "<字>"]`，按字號排序；不存官網 ID——開字詳情時會再查正向表。字號字母 A=常用、B=次常用、C=罕用、N=新增正字。
- `index.json`：`{"chars": [正向片首...], "codes": [反向片名(不含 code- 前綴)...]}`，前端查表前先比對，不存在的片連 fetch 都不發。

正向分片內一行一筆 JSON 陣列，按碼位排序：

```
["<字>", "<字號1>", <官網ID1>, "<字號2>", <官網ID2>, ...]
```

- `<字>`：字頭本身（BMP 之外的字是 surrogate pair）。
- `<字號>`：教育部字號。正字無連字號（如 `N00423`——PUA 字頭中有 17 個尚未編碼的新增正字）；異體字兩段式（如 `A00414-002`）；附字多為三段式。同一字可兼具多重身份（最多 4 個字號），依 mdx 原順序並列。
- `<官網ID>`：教育部官網頁面 ID，`https://dict.variants.moe.edu.tw/dictView.jsp?ID=<官網ID>` 可直達該字號的字形資訊頁（URL 格式出自 mdx StyleSheet 樣式 55 的連結模板；注意官網回應此頁時 HTTP 狀態碼是 404 但內容完整，瀏覽器顯示正常）。

用 `python3 extract_moe.py <mdx路徑>` 可從 mdx 重新產生（需 `pip install readmdict python-lzo`）。曾核對其中全部 41,521 個 PUA 字頭均存在於 `dt.jsonl`（同一套全宋體 PUA 編碼，零錯位）。
