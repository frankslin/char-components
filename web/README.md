# web/ — 部件檢索新實作

取代單一 `.htm` 檔案的新版前端。不用任何 bundler/建置工具——純 ES module + `fetch()`，部署到任何靜態網頁空間後，開啟 `web/index.html` 就能用。

```
web/
├── index.html   頁面骨架（查詢輸入、右側部件鍵盤/字詳情面板）
├── style.css    版面樣式（含深色模式、字源色彩）
├── core.js      比對演算法，逐句對照 legacy/部件檢索.htm 的核心程式碼翻譯而成
├── data.js      用 fetch() 載入 ./data/dt.jsonl、rt.jsonl、vt.json、kt.json(worker 側)與 bt.json(主執行緒側)
├── data/        dt/rt/vt/kt/bt 資料表本體(含 extract.py 與 schema 說明，見 data/README.md)
├── webfonts/    全宋體(WFG FSung) webfont 切片的 vendored 副本，供 style.css 用相對路徑載入
├── _headers     Cloudflare Pages/Netlify 的快取規則(字型切片長快取，wfg-fsung.css 短快取，.js 與 data/ 一律回源驗證——見下方部署注意)，做法比照 ../jigmo-webfonts
├── keypad.js    解析 kt.json 成 { icon, name, rows } 分類陣列、bt.json 成 { strokes, radicals } 筆畫陣列，供側邊鍵盤的兩種排列使用
├── blocks.js    字源(Unicode 區塊)分類的共用定義：色彩 class、圖例文字、篩選旗標字元
└── app.js       DOM 組裝：查詢輸入、即時搜尋、部件鍵盤、字詳情(含拆分樹渲染)、字源圖例、結果渲染、複製到剪貼簿
```

`web/` 除了字型之外沒有外部執行期相依：`data/` 是 vendored 進來的副本，程式碼零第三方套件。字型則由 `style.css` 開頭的 `@import` 指向 jsDelivr 上的 `wfg-fsung-webfonts@1.1.0`（版號鎖死）——分片修成真正的 woff2、整包降到約 80MB 之後才進得了 jsDelivr 的 150MB 上限；更早試過的 unpkg 因為代理 npm registry 的延遲不穩定而放棄。`web/webfonts/wfg-fsung/` 的 vendored 副本仍然保留、仍由 `webfonts/build-wfg-fsung.py` 的 `sync_vendor_copy()` 與 repo 根目錄那份（給 npm 套件用的）自動同步，要離線或內網部署時把那行 `@import` 換回 `./webfonts/wfg-fsung/wfg-fsung.css` 就是完全自足的版本。

## 為什麼不能用 `file://` 雙擊開啟

跟 `legacy/部件檢索.htm` 不同，這個新版**需要透過 http(s) 伺服器開啟**，不能直接雙擊本機檔案。原因是 `data.js` 用 `fetch()` 讀取 `./data/*`，多數瀏覽器在 `file://` 底下會擋掉這類同源請求。這是刻意的取捨：新實作的目標是「整體部署後，點開能用」，不是延續舊版「可離線雙擊」的約束。本機開發時起一個靜態伺服器即可：

```sh
python3 -m http.server 8000
# 開啟 http://localhost:8000/web/index.html
```

## 部署注意：模組之間不要有版本耦合

`app.js`／`worker.js`／`data.js` 是**分開下載**的 ES module，靜態空間或瀏覽器只要快取到其中一個的舊版，彼此的介面就對不起來。實際踩過一次：`bt`(部首筆畫鍵盤資料)原本跟 `dt`/`rt`/`vt` 一起在 worker 的 `init` 回傳，部署後使用者拿到新版 `app.js` 配舊版 `worker.js`(`init` 沒有 `bt`)，`parseRadicalPad(undefined)` 丟例外，整頁停在「資料載入失敗」——連查詢都不能用。

兩層防護：

1. **不要用 worker 的 `init` 當資料通道傳「UI 才需要的東西」。** `bt.json` 現在由 `app.js` 自己 `fetch()`(`data.js` 的 `loadRadicalTable()`)，跟 worker 無關；載入失敗最壞只是退回部件分類鍵盤，查詢照常。加新的 UI 資料表請照這個做法。
2. **`_headers` 讓 `/*.js` 與 `/data/*` 一律回源驗證**(`max-age=0, must-revalidate`，ETag 命中就是 304)。注意這只對吃 `_headers` 的平台有效(Cloudflare Pages／Netlify)，GitHub Pages 之類會忽略它——所以第 1 點才是主要防線。

## 功能

- **單一查詢輸入，拆字統一在字詳情面板**。輸入部件序列找字，點結果字塊即看該字的「字詳情」(右側面板，含拆分樹)。曾經做過「部件查字/拆字」兩個分頁與 `\字` 語法，後來發現都跟字詳情面板重複——查到字直接點就能看拆分——先後砍掉分頁與 `\字` 語法，統一由字詳情呈現(舊版 `?mode=tree` 分享連結仍相容)。
- **部件鍵盤(頁面級側欄)**：22 個分類(標點筆畫、字頭、包圍、部首類聚等，資料來自 `data/kt.json`，逐字沿用原版 `kt` 分類定義)**一次全部列出**，`position: fixed` 貼住視口右緣、佔滿整個視窗高度、內部捲動，主內容用右側 padding(`--sidebar-w`)讓出寬度。側欄內部**不分欄**：所有分類單欄從上到下排，每個分類是一條 flex-wrap 的「文字流」——行內小標籤 + 該分類全部按鍵攤平連排(原始 `kt` 的分列只是折行提示，這裡直接攤平以求密度)，按鍵縮小到剛好可讀，讓一屏能看到的部件數量最大化。點擊字元會插入到輸入框目前游標位置並觸發即時查詢；面板可收合(展開時只有「收起」鈕會收合，收起後整條可點擊展開)。窄螢幕(手機)改成固定在螢幕底部的抽屜：預設收起成一條、點開彈出半屏高鍵盤，按鍵加大到約 44px 觸控目標；觸屏裝置停用 hover tooltip，碼位資訊併入複製時的 toast。**手機上部首鍵盤與系統輸入法鍵盤互斥、共用同一條輸入框**：點部首鍵**不奪取焦點**(否則系統鍵盤會彈出來跟抽屜互相頂；無焦點時游標位置不可靠，改附加到末尾)；輸入框獲得焦點(系統鍵盤要彈出)時抽屜自動收起讓位；點收起條展開抽屜時輸入框自動失焦收掉系統鍵盤。字詳情的 ✕ 在手機上直接收起整個抽屜(桌面只切回鍵盤視圖)。
- **鍵盤有兩種排列，使用者自己選**：面板頂端的分段控件切換「依部首筆畫」(康熙 214 部首依畫數分組，資料在 `data/bt.json`，見 `data/README.md`；排左邊、**是預設**)與「WFG 漢字部件分類法」(上面那個 `kt` 分類鍵盤，原作者為這套拆分表訂的分類，只有繁體字形)。加第二種的原因是使用者回報「想要的部件不知道屬於哪個分類就找不到」(範例：「厷」)——部首表是紙本字典就熟悉的索引方式，畫數是客觀屬性不用猜；而且**部件不必精確**：拆分比對是遞迴展開的，用部首層級的部件照樣查得到字(實測「宀厶」查得到「宏」，宏＝宀＋厷、厷＝𠂇＋厶)，這點寫在切換鈕下方的說明裡。同一部首的正形與附形(人亻、水氵氺)包成一組不被折行拆散，附形用 muted 色與正形區隔；hover 顯示「部首 9　人（2 畫）」。**拆分表的筆形部件(㇒㇄𠃊…)也收進來**(34 個，排在部首後面、「筆形」小標籤之後)——它們不是部首但是拆分表最基本的葉節點、用量極大(㇒ 有 1273 個字用到)，只列 214 部首的話 1 畫組就只剩「一丨丶丿乙亅」六顆。**簡化部首歸在自己的畫數底下**(26 個，排在該畫數組末尾、前面加一個「簡化」小標籤)：「马」在 3 畫、「馬」在 10 畫，兩邊都找得到——掛在繁體正形旁邊當附形的話，簡體使用者在 3 畫組裡就是找不到「马」。切換鈕 `position: sticky` 貼在捲動容器頂端(鍵盤很長，捲到下面還要能換排法)。**選擇不做記憶**，重新整理回到預設的部首筆畫——`web/` 目前整個沒有 localStorage，見下方「刻意先不做」清單。
- **即時查詢(自動完成)**：預設開啟，輸入時debounce 120ms 後自動查詢(IME 組字中不會誤觸發)；即時查詢用較小的結果上限(500，精確命中不受限)，避免大量結果時卡頓；按「查詢」鍵或 Enter 一律跑完整上限(3000)。**比對計算在 Web Worker(`worker.js`)裡跑**：`getMatch()` 是同步計算，「同字異拆」開啟時單次可達數百毫秒，放主執行緒會讓打字/勾選卡頓(即時查詢每次 debounce 到期就凍一下)；`core.js` 純函式無 DOM 依賴，整個搬進 module worker 不需改動演算法。`dt`/`rt`/`vt` 與 matcher 只存在 worker 側，`app.js` 透過訊息 RPC(遞增 id 對應 Promise)取得渲染所需的純資料(hits/拆分樹/字源分類)，`kt`(部件分類鍵盤佈局)在 init 時一次交回；**`bt`(部首筆畫鍵盤)刻意不走這條路**，由 `app.js` 自己 `fetch()`——`app.js`/`worker.js` 是分開下載的模組，靠 init 傳鍵盤資料會讓兩者的介面版本綁在一起，部署後只要快取到舊版 `worker.js`(init 沒有 `bt`)就整頁「資料載入失敗」，實際發生過；現在 `bt.json` 讀不到最壞也只是退回分類鍵盤、查詢照常，切換鈕自動隱藏並在鍵盤區顯示一行說明。計數列顯示的秒數仍由 worker 內量測，只含純比對時間，口徑與先前一致。實測重查詢期間主執行緒最大凍結從 \~500-800ms 降到 \~30ms。結果超過上限時計數列顯示「超過 N 字」而非總數——`core.js` 沿用 legacy 的哨兵設計，模糊命中最多收 N+1 條、多的一條只用來偵測超限，此時列表長度(如 501)不是真實總數不能亮出來(移植時曾漏掉這個分支，恆顯示「總計 501 字」)。
- **偏旁口語名稱自動轉換**(`radicals.js`，165 個名稱)：輸入「三點水」自動變「氵」、「豎心旁」變「忄」——用注音/拼音打部首名稱比在鍵盤裡找部件快得多。簡繁拼寫都收；形狀類名稱(三點水/提手旁…)映射到偏旁變形部件，「X字旁」類映射到基礎字(靠預設開啟的「包容異體」摺疊變形)；竹字頭/雨字頭/人字頭等依 dt 實際用字映射到 𥫗/⻗/𠆢。轉換即時執行並以 toast 回饋；表中不收「是其他名稱前綴」的短別名(逐字打字時會提前吃掉前綴)，並有 Node 檢查驗證零前綴衝突。IME 組字中不觸發，整詞上屏後才轉換。**自動完成**：輸入尾端是名稱開頭(「三」「三點」⊂「三點水」)時，輸入框下方浮出候選清單(名稱+部件)，Tab 完成、↑↓ 選擇、Esc 關閉、點擊亦可；Esc 開著候選時先關候選、再按才清空輸入框。1 個字就開始提示(名稱總共才百來個)；但「木」「三」這類字本身就是合法部件，尾端只匹配 1 字時把「直接用這個字」放第一位當預設——Tab 只關候選不會被劫持成轉換，目標與原字相同的名稱項(「木字旁」→木)一併濾掉。
- **檢索碼位**：整個輸入若是碼位寫法——`U+4E00`/`u+f0200`/`0x2A6D6` 帶前綴，或 4\~6 位裸十六進位(`4E00`、`F0200`)——改走碼位查詢：命中已收錄字時列出該字的單一字塊，計數列顯示「碼位 U+XXXX → 「字」（分類）」；**確認查詢(Enter/查詢鈕/分享連結)還會直接打開字詳情**，即時查詢的逐鍵更新只顯示字塊不跳詳情(否則面板會隨打字亂切、手機抽屜會彈出來跟系統鍵盤互頂)。PUA 補充字的碼位也能查(這是外部字典都做不到的——PUA 沒有公開對照，只有本表的 `dt` 知道哪個碼位是哪個字)；部件保留列(⺀、㇀ 等)也可查，但 `dt` 下標 1\~10 的 `A`\~`J` 旗標佔位列會被排除(`U+0041` 不會查到假字「A」)。裸十六進位與部件查詢無衝突：十六進位的 ASCII 字元本來就不是部件，現狀只會查無結果。前綴後的 `+` 也接受空格——手打分享連結 `?q=U+4E00` 時 `+` 會被網址解碼成空格。查無此碼位時明確顯示「不在收錄範圍」，不會落回部件查詢出垃圾結果。另修 `core.js` `exhaust()` 一個繼承自 legacy 的堆疊溢位：拆分資料為佔位符「╳」的字(如多數無拆分資料的補充字)展開時，「╳」換算不出 `dt` 下標會誤拿 `dt[0]`(內容又含「╳」)無限遞迴——比照 `eliminate()` 的 `if (idx)` 防護改為視為不再分解(黃金案例回歸驗證通過，相對關係不變)。UI 層的穩定分區排序(`hit === 0` 整體提前，其餘維持拆分表原順序)，不動 `core.js` 的比對結果本身。
- **檢索字號**：整個輸入若是教育部《異體字字典》字號——裸正字號(`A02353`)或帶後綴的異體/附字號(`A02353-004`，後綴須照官方零埋 3 位；字母 A=常用、B=次常用、C=罕用、N=新增正字，大小寫皆可)——改走字號反查(`web/data/moe/` 的 `code-*` 反向分片，按需載入)：裸正字號列出**整個字族**(正字＋全部異體/附字，計數列顯示「字號 A02353 → 「濱」字族共 N 字」)，帶後綴字號只列該字；確認查詢同樣直接打開字詳情(字族開正字的詳情)。與裸十六進位碼位查詢**零衝突**：`[A-C]` 開頭加 5 位數字若當十六進位讀必然大於 `0x10FFFF`(`parseCodepointQuery()` 回 null 放行)，`N` 不是十六進位字母。少數字族(24/29920)裡同一字掛兩個字號(官方同形字)，字塊去重顯示。字詳情面板裡凡是查得到教育部字號的字(含 PUA 補充字)都有**「查異體字」按鈕**(主色填充，與外部字典的外框連結區隔)：取字號 dash 前的正字號塞回輸入框重查，即列出整個字族並打開正字詳情；同一字掛多個字族時每個字族一顆按鈕。**標籤帶上該字族的正字**——寫成「查異體字「壘」A00825」，光看字號沒人認得會查到什麼；正字要另查反向字號表(`findMoeCode`)、非同步，先用純字號當佔位標籤、查回來再補正字(補之前用 `lastDetail` 確認使用者還停在同一字)。**點按鈕算一次新導覽**：跳去查字族前先把 `urlSettled` 設回 true，讓這次跳轉走 `pushState`，「上一頁」能退回原本的查詢——使用者常是即時查詢(未定案、`urlSettled=false`)途中點進詳情的，不先標記定案就會 `replaceState` 蓋掉原查詢、退不回去。查無時明確顯示「字號 XXX 查無此字」。
- **字源顏色**：查詢結果的字塊用低飽和的字源(Unicode 區塊)淺色底 + 同色系虛線框表達分類；「基本」區佔絕大多數結果，維持中性底色，避免整片結果變成彩虹。圖例直接展開顯示在結果下方，**純顏色說明、不可點擊**——所有區塊一律同時查詢，不提供分區篩選 UI。命中類型(精確/模糊/正體)只用文字顏色/字重表達，不與字源底色互相干擾。hover 字塊會**立即**顯示 `U+` 碼位與分類(自製 tooltip 浮層，不用原生 `title`——原生的有約一秒的內建延遲且無法調整)。
- **「相容」區一律不列出**：「相容表意文字」都是外觀與基本區重複的相容碼位，一般使用者不該把它們打進文件，所以 `app.js` 在使用者沒有自己指定篩選符號時，自動在查詢前面補上「除了相容以外」的區塊旗標(`DEFAULT_FLAGS`)。`core.js` 的旗標語法(`@`/`A`\~`J`/`X`/`Y`/`Z`)仍然有效，真的要查相容區可以手動在輸入前打 `X`，但 UI 上不再提供入口。
- **補充字(PUA)提示**：「補充」分類的字使用作者暫定的私有造字區碼位，不是正式 Unicode 編碼，hover 這類字塊只會顯示「補充（尚未正式編碼，暫用私有碼位）」，不會顯示假的 `U+` 碼位；其餘已正式編碼的字，hover 會顯示 `U+XXXX · 分類名稱`。
- **包容異體**、**同字異拆(顯示全部拆法)**、**排除未編碼補充字**：三個核取方塊，語意對應 `core.js` 的 `v`/`d`/`u` 參數。「包容異體」**預設勾選**——拆分表大量使用偏旁變形(𠆢/亻/⺡…)，不開的話「石人工」查不到「砼」(拆分是「石、𠆢、工」)這類情況對一般使用者來說就是「查不到」，關掉屬於進階用法。「同字異拆」也**預設勾選**(與 legacy 版 cookie 預設 `subdivide=1` 一致)——`dt` 裡有一批字(如 `风!𠘨㐅`、`羽!习习`)第一分支為空、拆法只存在於 `!` 分支，不開的話這些字用部件反查永遠查不到(輸入 `𠘨㐅` 查不到「风」)，性質同「包容異體」。代價是 `eliminate()` 要走訪的分支變多，單次查詢實測從約 180ms 增至約 700ms，可接受；曾短暫預設關閉，屬移植時未記載的行為回歸，已改回。
- **點字 = 複製 + 字詳情**：點結果字塊會複製到剪貼簿，並把右側面板切到「字詳情」頁籤，**一次列出**該字的全部資訊，不需要再點按鈕展開(曾經做過 hover 浮現的小工具列——複製/查字典——後來移除：點擊本來就複製，查字典在詳情面板有更完整的入口，工具列冗餘)：大字、`U+` 碼位/字源、**外部字典大按鈕放頂部**(字統網、漢典、教育部異體字字典、國學大師、萌典、維基詞典中/英/日、Unihan、小學堂字形演變、CTEXT、GlyphWiki——Unihan 是 Unicode 官方資料庫，再罕見的正式編碼字都有記錄；GlyphWiki 以小寫十六進制碼位定址；國學大師的 URL 格式出自其官方說明頁，站點有地區連線限制未能直接實測)、拆分樹。**PUA 補充字沒有正式碼位，一般外部字典查不到**，但其中 41,521 字是《教育部異體字字典》的字頭——詳情面板會非同步查 `web/data/moe/` 字號對照表(見 `web/data/README.md`)，查到就列出「教育部異體字字典 A00414-002」樣式的字號直鏈(`dictView.jsp?ID=…` 直達官網該字號頁面；官網回應此頁時 HTTP 狀態碼是 404 但內容完整，瀏覽器顯示正常)，一字兼多重身份(正/異/附字)時逐一列出；查不到才以一行說明代替連結。已編碼字也受惠：查到字號時，把泛用的「以字搜尋」教育部連結原地換成字號直鏈，比搜尋更精準。對照表按需分片載入，開詳情前不佔流量；載入前先掛佔位容器，回來後用 `lastDetail` 檢查使用者是否已點開別的字，是則丟棄，避免舊結果污染新詳情。「部件鍵盤／字詳情」頁籤做成**分段控件**(選中主色填充、未選中維持完整可點按外觀，手機上撐滿抽屜頂部、近 44px 觸控目標)——曾經是 12px 灰字幽靈按鈕，看起來像說明文字，手機上點開字詳情後使用者發現不了還能切回鍵盤(帶 `?q=U+…`/字號的分享連結一進來就是字詳情視圖，更嚴重)。面板若處於收起狀態會自動展開；詳情右上角有 ✕ 可關閉，關掉後回到部件鍵盤；輸入框重新獲得焦點(要繼續組部件)時也會自動切回部件鍵盤。鍵盤與字詳情共用同一個捲動容器(`panel-body`)，每次顯示新詳情都會把它捲回頂部——不然若使用者剛把鍵盤捲到很下面，新詳情的頂部(關閉鈕/大字)會被捲出畫面外，尤其在手機的底部抽屜上特別明顯(抽屜矮，捲動位置殘留更容易讓人以為「沒反應」)。**拆分畫成真正的自上而下節點樹**(org-chart 風格，巢狀 ul/li + CSS 連接線)：`exhaust()` 輸出的括號嵌套文字(`咅(立(亣一┇…)口)⻏`)在 UI 層解析成節點樹，根節點是字本身；內層「┇」的一字多拆分支插入虛線圓的「或」節點分隔。拆分樹尊重「同字異拆」選項(對應黃金案例 `\主` 的行為)：未勾選只畫第一種拆法並提示還有幾種，勾選後全部拆法**左右並排**，容器可橫向捲動。
- 點擊結果字塊會複製到剪貼簿並顯示提示(toast 帶 `U+` 碼位與字源分類)。
- **字型策略(系統字型優先)**：漢字的字型棧是「系統宋體/明體(Songti TC/SimSun/PMingLiU/Noto Serif…) → WFG FSung webfont → serif」——常用碼位直接用系統字型**零下載**，系統字型缺字的罕用擴充區字由瀏覽器逐字回退到 FSung 的 256 碼位 WOFF2 切片(用到才下載)。**補充字(PUA)例外**：已知是 PUA 碼位的元素(結果字塊/拆分樹/鍵盤按鍵)都掛 `.pua-char` 強制 FSung 優先，因為個別系統字型的 BMP PUA 碼位可能撞車顯示成不相干的字(AGENTS.md 約束 5；資料中 PUA 絕大多數在 15/16 輔助平面，僅 1 字在 BMP PUA)。注意**不能**用 `@font-face src: local()` 做範圍控制——Chrome 86+ 基於反指紋考量不查詢本地字型，實測 headless/真實 Chrome、WebKit 都不匹配(僅 Firefox 部分可用)。實測：只查基本區(`@日月`)零切片下載；含補充字的查詢會正確下載對應 PUA 切片。
- **字型切片的分波載入與 Chrome 豆腐字 workaround**：帶 `?q=` 直開分享連結時，結果一次渲染會讓瀏覽器同時對幾百個字型切片(平均約 170KB/片)發請求，總量可達幾十 MB，排最前面的精確命中(往往正是只有全宋體才有字形的擴充區/補充字)反而要跟所有切片搶帶寬。`renderHits()` 因此**分波渲染**：每波 240 個字塊(約蓋滿第一屏)，等上一波觸發的字型下載完(輪詢 `document.fonts.status`，至多等 1 秒)再放下一波——結果前面的字永遠優先拿到帶寬；快取全熱時各波幾乎連續，感受不到分批。結果區的清空統一走 `clearOutput()`(遞增 renderToken 作廢排程中的波次)，避免清除/新查詢後舊波次的字塊又冒出來。另有一個 **Chrome 缺陷 workaround**(實測 Chrome/macOS，症狀：帶 `?q=` 直開時前幾個罕見字永久顯示 .notdef 方框、反復刷新無效，手動輸入同一查詢卻正常)：文字排版時所需切片還沒下載完的話，Chrome 用回退鏈裡系統有的字族(Songti TC)畫方框佔位，切片到貨後的重排並不可靠，而且中毒的是 Blink 按「字族鏈+字號」快取的回退解析結果——同一字號下重建文字節點、切換 font-variant 都救不活，只有改字號(換一把快取鍵)能強迫重新解析。所以 `style.css` 給所有顯示資料字的 `font-size` 乘上 `--font-nudge` 係數，`app.js` 在每批字型下載完成(`fonts` 的 `loadingdone` 事件)時換一個新值：步長 0.002(低於此會被 Blink 約 1/64px 的字號快取鍵量化吞掉)、十個值單調輪轉且永不回到 1(每個「下載期間排過版」的值都可能中毒，翻回去就復發)，最大偏差 2% 肉眼與版面無感。WebKit/Firefox 無此缺陷，workaround 對它們無副作用。
- **可分享連結**：查詢內容與狀態同步到 URL query string——`q`(輸入)、`v=0`/`d=1`/`u=1`(偏離預設值的選項才寫入)、`c=聚焦的字`；舊版含 `mode=tree` 的連結仍相容(自動打開該字的字詳情)。開啟帶參數的連結會還原全部狀態並直接執行查詢。**`c` 記錄「正在看哪個字」**：用部件查到一堆字後點某個字塊看詳情，網址會補上 `c=該字`(如 `?q=日月&c=明`)並 `pushState` 一筆歷史，分享出去對方會同時看到「怎麼查的」與「在看哪個字」、開啟時自動打開那個字的詳情。碼位/字號查詢本身就指明了字，`c` 不重複帶。每點一個字塊各是一筆歷史，「上一頁」依序退回先前聚焦的字、再退回純結果列表(此時詳情面板收起、切回鍵盤)；按詳情的 ✕ 用 `replaceState` 去掉 `c`(不新增歷史，免得「上一頁」又把剛關掉的詳情叫回來)；打新查詢會清掉 `c`。歷史棧策略：「按查詢/Enter 確認過」「點字塊聚焦」與「由連結/前後退還原」的狀態視為定案(settled)紀錄，之後第一次變更先 `pushState` 保護它，連續變更(即時查詢逐鍵)只 `replaceState`——打一個詞不會塞進十筆歷史，但「後退」永遠回到上一個定案狀態。即時查詢(未定案)途中點字塊聚焦時會先把狀態標記為定案，確保這次聚焦走 `pushState`、原查詢留在歷史棧(同「查異體字」按鈕)。

## 跟 legacy 版比，目前刻意先不做的功能

- 沒有「異體檢索」入口（`core.js` 已經有 `getVariant()`，邏輯已就緒，只是 `app.js` 還沒接 UI）。
- 沒有 cookie/`localStorage` 記住使用者的選項狀態(每次重新整理都會回到預設值)。
- 原版的「跳轉外部字典」已接上(固定用 zi.tools，不像原版 `ref` 可客製化連結)，見上面「點字行為選擇＋字詳情面板」。
- `data/` 幾個檔案（共約 4MB）會一次整包載入，還沒做成 `webfonts/wfg-fsung` 那種按需分片。

這些都是後續可以再加的功能，不是遺漏或 bug；`core.js` 的比對邏輯本身已經是完整、經測試驗證的核心。
